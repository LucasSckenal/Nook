# 09 — Estrutura da IA ("Estuda")

A IA do Nook tem nome, personalidade e limites claros. **Estuda** é uma colega de quarto que entende o semestre inteiro do aluno — não um chatbot genérico com um textarea.

## 1. Princípios

1. **Contexto estruturado > prompt esperto.** O diferencial não é o modelo, é o que ele sabe: disciplinas, fórmulas de nota, carga, ritmo real do aluno.
2. **Propõe, nunca executa sozinha.** Todo artefato (plano, flashcards, reagendamento) exige confirmação explícita. A agência é do estudante.
3. **Gentileza é restrição técnica.** O system prompt proíbe culpa, alarme e comparação social — mesma política do design system.
4. **O produto funciona 100% sem IA.** Estuda é aceleradora, não dependência. (Resiliência + custo.)
5. **Custo sob controle.** Cache agressivo, modelos por tarefa, cotas por plano.

## 2. Arquitetura

```
Cliente (web/app)
   │  HTTPS callable (App Check + Auth)
   ▼
aiGateway (Cloud Function) ───────────────────────────────┐
   │ 1. valida cota do usuário (plan: free/plus)          │
   │ 2. CONTEXT BUILDER: monta o AcademicContext          │
   │    a partir do Firestore (só o necessário p/ intent) │
   │ 3. roteia o modelo por tarefa                        │
   │ 4. chama o LLM com tools definidas                   │
   │ 5. valida o artefato retornado (schema Zod)          │
   │ 6. grava AIMessage + artifact; nunca grava em        │
   │    tasks/calendar sem flag applied=true              │
   ▼
LLM (API Claude)
   • conversas e planos: claude-sonnet-4-6
   • flashcards/resumos em lote, classificação: claude-haiku-4-5
   • ingestão de plano de ensino (PDF/imagem): sonnet com visão
```

### Roteamento por tarefa

| Tarefa | Modelo | Latência alvo | Custo relativo |
|---|---|---|---|
| Chat e planejamento | Sonnet | streaming | médio |
| Resumo de anotação | Haiku | < 4s | baixo |
| Geração de flashcards | Haiku | < 6s | baixo |
| Leitura do plano de ensino | Sonnet (visão) | < 20s, 1×/disciplina | médio |
| Detecção de risco (batch noturno) | heurística + Haiku p/ redação do aviso | offline | mínimo |

## 3. AcademicContext (o coração)

O Context Builder monta um JSON compacto e **escopado por intent** — nunca despeja o banco inteiro no prompt:

```jsonc
{
  "student": { "name": "Marina", "course": "Psicologia",
               "studyWindow": "19:00–23:00",
               "avgSessionMin": 32, "bestHours": "20–22h" },
  "calendar": {
    "now": "2026-06-11T22:00-03:00",
    "freeSlots7d": [ ... ],            // derivado: janela de estudo − aulas − planos
    "upcoming": [
      { "type": "exam", "subject": "Anatomia", "in_days": 2, "weight": 0.35 },
      { "type": "assignment", "subject": "Física", "in_days": 6 }
    ]
  },
  "subjects": [
    { "name": "Cálculo II", "partialGrade": 6.2, "target": 7.0,
      "neededOnNext": 7.4, "focusMin14d": 50, "risk": 0.42,
      "openTasks": 3 }
  ],
  "workloadRadar": { "w24": "light", "w25": "moderate", "w26": "heavy" },
  "recentBehavior": { "sessions7d": 5, "planAdherence": 0.7,
                      "skippedPlanDays": ["2026-06-09"] }
}
```

Escopos: `chat_general` (tudo resumido) · `subject_focus` (1 disciplina em detalhe) · `planning` (calendário + slots) · `content` (anotação/material em questão).

## 4. Tools expostas ao modelo (function calling)

```
propose_study_plan(subjectId, examId?, sessions[{date, start, minutes, topic}])
propose_reschedule(moves[{taskId|sessionId, from, to}], reason)
generate_flashcards(noteId|topic, cards[{front, back}])
generate_summary(noteId|materialId, style: "outline"|"prose"|"cornell")
flag_risk(subjectId, severity, factors[], suggestion)
query_context(scope)            ← pedir mais contexto sob demanda
```

Toda tool retorna **proposta** → o gateway valida com schema → o cliente renderiza o `AIArtifact` com botão de confirmação → só então grava nas coleções reais (com `source: "ai_plan"` para auditabilidade e desfazer em lote).

## 5. System prompt (essência, resumida)

```
Você é a Estuda, colega de estudos do Nook. Você conhece o semestre
do estudante pelo AcademicContext anexado — use-o; não invente dados.

TOM: calma, calorosa, direta. Português brasileiro coloquial-leve.
Frases curtas. Você está do lado do estudante, nunca acima dele.

PROIBIDO: culpar ("você deveria ter..."), alarmar ("URGENTE!"),
comparar com outros, prometer nota, dar diagnóstico de saúde mental
(acolha e sugira apoio profissional se o tema surgir).

PLANEJAMENTO: respeite freeSlots e a studyWindow; sessões de 25–50min;
nunca planeje mais de 70% do tempo livre (folga é parte do plano);
se o aluno furou dias do plano, replaneje sem comentar a falha.

AÇÕES: para criar/alterar qualquer coisa, use as tools e apresente
como proposta. Nunca afirme que algo foi criado antes da confirmação.
```

## 6. Capacidades, uma a uma

### 6.1 Criar cronogramas
Entrada: prova/entrega alvo + `freeSlots`. Algoritmo híbrido: a heurística (espaçamento, interleaving, folga de 30%) gera o esqueleto; o LLM distribui tópicos e redige. Saída: `propose_study_plan` → blocos fantasma no calendário → confirmação.

### 6.2 Gerar resumos
Da anotação/material selecionado. Estilos: tópicos, prosa, Cornell. O resumo vira `aiArtifact` anexado à nota original (rastreável, refazível).

### 6.3 Criar flashcards
De anotação, resumo ou tópico. Saída validada (máx. 30 cartões/lote, frente ≤ 200 chars). Entram no deck com SRS zerado; revisões aparecem no Dashboard ("12 cartões para hoje").

### 6.4 Planejar revisões
Agenda automática pós-aprendizado (1d, 3d, 7d, 21d) combinando decks SRS + sessões curtas no calendário. Estuda sugere a ativação por disciplina, uma vez — sem insistir.

### 6.5 Detectar riscos acadêmicos
Função batch noturna, **heurística primeiro** (transparente e barata):

```
risk = w1·(gradeGap) + w2·(timeUnderInvested) + w3·(upcomingLoad) + w4·(lowAdherence)
```

- `gradeGap`: média parcial vs. nota necessária no que falta (usa `gradeFormula`)
- `timeUnderInvested`: foco na disciplina vs. média pessoal
- `upcomingLoad`: colisões no radar de 4 semanas
- `lowAdherence`: planos furados recorrentes

risk > 0.6 → Estuda redige **um** aviso gentil com proposta concreta ("Cálculo precisa de ≥7,4 na P2. Duas sessões a mais por semana resolvem — monto?"). Máximo 1 alerta de risco/dia, nunca empilhar.

## 7. Guardrails e privacidade

- **App Check + Auth** no gateway; cliente nunca fala com o LLM direto; chave da API só na Function.
- Cotas: free = 20 interações/dia + 3 ingestões de documento/mês; Plus = generoso com rate-limit anti-abuso.
- Dados enviados ao LLM: mínimo necessário, sem e-mail/identificadores; flag de "não treinar" contratual; opt-out de IA nas configurações desliga tudo (produto segue inteiro).
- Saída validada por schema; payload que não valida → retry 1× → fallback gracioso ("não consegui montar agora, tenta de novo?").
- Conversas retidas 90 dias (free) com expurgo; usuário pode apagar threads.

## 8. Métricas da IA

| Métrica | Alvo |
|---|---|
| % de propostas confirmadas (plan acceptance) | > 60% |
| Aderência aos planos criados | > 50% das sessões cumpridas |
| Tempo de setup via ingestão de plano de ensino | < 10 min |
| Custo de IA / usuário ativo / mês | < US$ 0,40 (free), < US$ 2 (Plus) |
| Taxa de fallback (IA indisponível) | < 1% das interações |
